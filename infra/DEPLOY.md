# Deployment Guide

## Overview

A single EC2 t4g.small instance runs the Node.js backend and PostgreSQL. Nginx terminates TLS and proxies to Node on port 3000. Express serves both the API and the Vite-built frontend. Postgres data lives on a separate EBS volume that survives instance replacement.

Deployments are triggered by pushing to `main`. GitHub Actions builds the app, uploads a tarball to S3, then runs a deploy script on the instance via AWS SSM Run Command — no SSH keys required anywhere.

```
GitHub push → Actions build → S3 artifact → SSM → instance pulls & restarts
```

---

## Before you begin

Adapt the following to your own environment before deploying:

| What | Where to change |
|---|---|
| AWS account ID and region | `infra/bin/app.ts`, `npx cdk bootstrap` command |
| Domain name and Route 53 hosted zone ID | Constants at the top of `infra/src/stacks/EC2Stack.ts` |
| GitHub repository name | `token.actions.githubusercontent.com:sub` condition in `EC2Stack.ts` |
| S3 bucket name (includes account ID) | `env.S3_BUCKET` in `.github/workflows/deploy.yml` |

## First-time setup

### 1. Bootstrap CDK (once per AWS account/region)

```bash
cd infra
npm install
npx cdk bootstrap aws://777312966064/us-east-1
```

### 2. Deploy the stack

```bash
npx cdk deploy
```

Note the outputs — you'll need them in step 5:

| Output | Used for |
|---|---|
| `OutDeployRoleArn` | GitHub secret `AWS_DEPLOY_ROLE_ARN` |
| `OutInstanceId` | GitHub secret `EC2_INSTANCE_ID` |
| `OutElasticIP` | Verify DNS propagation |

### 3. Wait for user data to finish

The instance bootstraps itself on first boot (installs Postgres, Node, Nginx, PM2). This takes 3–5 minutes. You can watch progress via SSM Session Manager:

```bash
aws ssm start-session --target <instance-id>
sudo tail -f /var/log/user-data.log
```

### 4. Create the app environment file

SSH or SSM into the instance and create the backend `.env`:

```bash
sudo -u ec2-user mkdir -p /opt/family-tree/packages/backend
sudo -u ec2-user tee /opt/family-tree/packages/backend/.env << 'EOF'
DATABASE_URL=             # copy from /opt/family-tree/db.env (generated by user data)
JWT_SECRET=               # generate: openssl rand -base64 32
ADMIN_EMAIL=              # your email address
ADMIN_PASSWORD=           # strong password for the admin account
APP_URL=https://tree.slashdave.com
AWS_REGION=us-east-1
SES_FROM_ADDRESS=         # your verified SES address — required to send invite emails
PORT=3000
EOF
```

`DATABASE_URL` is already generated and stored at `/opt/family-tree/db.env` by user data.

### 5. Add GitHub secrets

In the repo → Settings → Secrets and variables → Actions, add:

| Secret | Value |
|---|---|
| `AWS_DEPLOY_ROLE_ARN` | `OutDeployRoleArn` from CDK output |
| `EC2_INSTANCE_ID` | `OutInstanceId` from CDK output |

### 6. Issue the TLS certificate

Wait for DNS to propagate (check with `dig +short tree.slashdave.com`), then on the instance:

```bash
sudo certbot --nginx -d tree.slashdave.com \
  --non-interactive --agree-tos \
  -m admin@slashdave.com
```

Certbot modifies the Nginx config and installs a renewal cron automatically.

### 7. Trigger the first deploy

Push any commit to `main` (or re-run the workflow manually in Actions). The pipeline will build the app, deploy it to the instance, run migrations, and start PM2.

### 8. Seed the database (first time only)

Compile and copy the seed script to the instance, then run it:

```bash
# From your local machine (in the repo root)
npm run build -w packages/backend
scp packages/backend/dist/scripts/seed.js ec2-user@<elastic-ip>:/tmp/seed.js

# On the instance (via SSH or SSM session)
cd /opt/family-tree/packages/backend
node /tmp/seed.js
rm /tmp/seed.js
```

The seed script uses the `DATABASE_URL` already set in `.env`.

## Ongoing deployments

Push to `main`. The workflow handles everything:

1. Lint + build (on the GitHub Actions runner)
2. Upload tarball to `s3://family-tree-backups-777312966064/deploy/`
3. SSM Run Command: download, extract, `npm ci`, `prisma migrate deploy`, `pm2 restart`

Typical deploy time is 2–3 minutes.

## Restoring Postgres from backup

**From EBS snapshot** (preferred — restores the entire volume):

1. Stop the instance
2. Detach the data volume (`/dev/sdf`)
3. Create a new volume from the desired snapshot
4. Attach to the instance as `/dev/sdf`
5. Start the instance

**From weekly pg_dump** (surgical restore):

```bash
# List available backups
aws s3 ls s3://family-tree-backups-777312966064/weekly/

# Download and restore
aws s3 cp s3://family-tree-backups-777312966064/weekly/2026-05-25.sql.gz /tmp/
gunzip -c /tmp/2026-05-25.sql.gz | psql -U familytree family_tree
```

## Useful commands

```bash
# Check app status
runuser -l ec2-user -c "pm2 status"
runuser -l ec2-user -c "pm2 logs family-tree --lines 50"

# Restart app
runuser -l ec2-user -c "pm2 restart family-tree"

# Nginx
sudo nginx -t && sudo systemctl reload nginx

# Postgres
sudo systemctl status postgresql
sudo -u postgres psql
```

## Stack resources

| Resource | Detail |
|---|---|
| EC2 | t4g.small, ARM64, Amazon Linux 2023 |
| Root EBS | 20 GB gp3, deleted on termination |
| Data EBS | 20 GB gp3, **RETAIN** — Postgres data |
| EBS snapshots | Daily at 03:00 UTC, 7-day retention (AWS DLM) |
| pg_dump | Weekly to S3, 90-day retention |
| Elastic IP | Stable across stops/restarts |
| DNS | `tree.slashdave.com` → Elastic IP |
